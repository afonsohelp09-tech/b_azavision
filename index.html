🌐 Idioma e moeda (vitrine)

A maioria dos utilizadores fala português: este painel e a vitrine usam português (PT) como língua por defeito. O seletor no topo (PT · FR · EN · ES) e o da loja oferecem sempre francês, inglês e espanhol — línguas que devem estar disponíveis para clientes internacionais.

Primeira visita na loja: aplica o idioma escolhido abaixo (recomendado: Português). No cabeçalho da vitrine o cliente pode alternar sempre entre PT, FR, EN e ES.

💳 Vitrine — moyens visibles au checkout
Pour vos clients : cochez uniquement les moyens que vous acceptez réellement à l’étape paiement. Ce panneau règle l’affichage sur la boutique ; les connexions réelles (Stripe, IBAN, etc.) se configurent dans l’onglet Paiements. Si vous modifiez le fichier api_apps_script.gs dans Google Apps Script, redéployez la Web App (« Déployer » → « Gérer les déploiements » → nouvelle version) et mettez à jour l’URL API dans l’admin si besoin.
  1. Onglet Paiements : clés Stripe / PayPal / IBAN selon vos besoins, puis Enregistrer les paiements.
  2. Ici : cocher les moyens visibles au checkout et enregistrer.
  3. Tester sur la vitrine (mode test Stripe si pk_test_).
  4. Production : pk_live_ + clé secrète sk_live_ dans les propriétés du script.
Règles utiles
  • Stripe et Cartão : si vous cochez « Utiliser Stripe pour l’option Cartão », le flux carte passe par Stripe ; sinon « Cartão » peut rester en démo ERP selon votre réglage.
  • MB Way, Multibanco, PayPal (cases seules) : l’affichage ne suffit pas pour encaisser automatiquement — prévoir intégration PSP ou développement (voir onglet Paiements).
  • Virement / COD : pas de prélèvement en ligne ; vous validez la commande après contrôle manuel.

💳 Paiements — connexion aux services (clés & comptes)
Guide intégrateur / client : cette page relie les services réels (Stripe, IBAN…). L’affichage des logos / options au checkout se règle dans Boutique → « Vitrine — moyens visibles ». Les deux vont ensemble : sans case cochée en vitrine, le client ne voit pas le moyen ; sans clés ici, le paiement en ligne ne peut pas aboutir.
  1. Renseigner ci‑dessous les identifiants nécessaires, puis Enregistrer les paiements.
  2. Aller dans Boutique et activer les mêmes moyens côté vitrine.
  3. Clé secrète Stripe uniquement dans Apps Script → propriété STRIPE_SECRET_KEY (sk_test_ ou sk_live_), jamais dans Google Sheets ni dans le HTML client.
  4. Après modification de api_apps_script.gs : nouveau déploiement de la Web App et vérifier que l’URL API dans l’admin est à jour.
  5. Déploiement Web App : accès Toute personne ayant le lien (ou équivalent) pour que la boutique puisse appeler l’API sans erreur d’authentification.

⚡ Stripe — cartes, Apple Pay, Google Pay

Un seul bloc Stripe couvre la plupart des paiements par carte et portefeuilles. Les boutons Apple/Google apparaissent si le compte Stripe, le domaine et le navigateur le permettent.

  1. Créer / se connecter à dashboard.stripe.com — commencer en mode test (interrupteur « Test » en haut à droite).
  2. Développeurs → Clés API : copier la clé publique pk_test_... ou pk_live_... dans le champ ci‑dessous.
  3. Même onglet : copier la clé secrète sk_... dans le projet Extensions → Apps Script → ⚙️ Propriétés du script → clé STRIPE_SECRET_KEY (la valeur commence par sk_).
  4. Les clés pk_ et sk_ doivent être du même mode (toutes deux test ou toutes deux live).
  5. Stripe → Paramètres → Moyens de paiement : activer cartes, Link, wallets selon votre offre et pays.
  6. Enregistrer les paiements dans l’admin, puis tester un panier : carte test 4242 4242 4242 4242, date future, CVC quelconque (mode test uniquement).
Production / test
Passage en production

Remplacer pk_test_ par pk_live_, mettre à jour STRIPE_SECRET_KEY avec sk_live_, redéployer la Web App si le script a changé, puis refaire un achat réel de contrôle (petit montant).

🅿️ PayPal

L’ERP peut stocker un Client ID pour une intégration PayPal (Checkout / Smart Buttons) à brancher dans le code serveur et la boutique. Sans ce développement, cocher « PayPal » en vitrine n’encaisse pas automatiquement.

  1. Compte developer.paypal.com → My Apps & Credentials.
  2. Créer une app (Sandbox pour les tests, Live pour la production).
  3. Copier le Client ID (format alphanumérique long) dans le champ ci‑dessous.
  4. Secret / certificats : côté serveur uniquement (Apps Script ou backend), jamais exposés au navigateur.
Intégration à finaliser

📱 MB Way & Multibanco (Portugal)

Ces méthodes sont souvent proposées par un prestataire local ou par Stripe selon votre compte. Dans ce projet, les cases vitrine servent surtout à informer le client ; l’encaissement automatique nécessite un contrat PSP et du développement (API banque / agrégateur).

À prévoir côté métier
  • MB Way : convention avec la banque ou passerelle ; URL / QR de paiement gérés par le PSP.
  • Multibanco : référence générée par le prestataire (SIBS, etc.) ; le client paie en bureau de poste / appli banque.
  • Vérifiez dans Stripe si ces moyens sont disponibles pour votre pays et votre compte avant de les promettre sur la boutique.

🏦 Virement bancaire

Les clients paient hors ligne ; vous vérifiez la réception manuellement avant d’expédier. L’IBAN saisi ici est utilisé pour l’affichage / consignes côté commande.

Manuel

📦 Paiement à la livraison

Aucune clé : cocher pour autoriser le paiement en espèces ou CB au livreur. Les commandes apparaissent dans l’admin pour traitement comme les autres ; vous confirmez la réception des fonds hors ligne.

Hors ligne
Dépannage rapide
  • Erreur réseau / CORS : l’URL de l’API doit être celle du déploiement Web App Apps Script (pas l’URL d’édition du script).
  • 403 / accès refusé : vérifier que la Web App est déployée pour « Toute personne » (accès anonyme) si la boutique est publique.
  • Stripe ne charge pas : STRIPE_SECRET_KEY absente ou mode test/live mélangé avec la clé publique.
  • Rien ne change sur la boutique : enregistrer dans l’admin, vider le cache (Ctrl+F5), vérifier vitrine + onglet Paiements.
